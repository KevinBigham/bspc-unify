## ¬ß0 Why the tripwire fired

The Phase F lesson ("audit every do-nothing default for auth-cutover coupling")
found the notification tier is MOSTLY do-nothing defaults, and the ones that
matter are unpinned:

1. **The evaluator's firing mechanism is unpinned.** `evaluateNotificationRules`
   is a Firestore trigger on `attendance/{id}` ‚Äî dark since Phase C (ratified),
   dead forever at cutover (the F RF-2 class). D-F2 (client-invoke + sweeper)
   was ratified FOR MEDIA; extending it to attendance is presumed but not pinned,
   and the kick payload/sweep design is new surface.
2. **Coach push delivery has no pinned transport ‚Äî and a seam.** Today's chain
   (notifications doc ‚Üí `onNotificationCreated` ‚Üí FCM to `coaches.fcmTokens`)
   dies with Firestore. The canonical transport (BSPC's `send-notification`
   Edge Function) **unconditionally inserts `in_app_notifications` rows for
   every job target** ‚Äî any design where Coach writers both write in-app rows
   (required for the idempotency UNIQUE) and enqueue jobs double-notifies.
   And the Deno sender has zero tests (RG-5), so patching it can't meet our bar.
3. **OD-4 is explicitly open** (banked from Phase A: dailyDigest enumeration
   source), and the digest pref (`coaches.notificationPrefs.dailyDigest`,
   written default-true by AuthContext at signup, no toggle UI anywhere) has
   **no canonical home** ‚Äî `notification_preferences` has only `push_enabled`.
4. **`ai_drafts_ready` routing is "G's whole"** (banked from F) ‚Äî the category
   exists only as a declared type; no writer has ever existed. Wiring it is new
   behavior; G must decide, not assume.
5. **Nothing anywhere schedules the canonical pipeline.** `send-notification`
   and `cleanup-tokens` have no invoker ‚Äî no cron, no caller, no pg_cron. The
   entire BSPC notification feature (including its in-app rows, which the
   sender creates) is dormant pending an invocation decision. Pure
   auth-cutover-coupled do-nothing default.
6. **Cross-phase finding (RG-6):** no migration has ever added a table to the
   `supabase_realtime` publication. Every Coach realtime subscription landed in
   B‚ÄìF (12 tables) plus G's two depends on hosted publication config that
   exists nowhere in code.

## ¬ß4 Decisions

**D-G1 ‚Äî Evaluator invocation.**
(a) **RECOMMENDED ‚Äî extend D-F2 verbatim: data-layer kick + scheduled sweep.**
`checkIn`/`batchCheckIn`/`checkOut` (the Firestore trigger fired on every
attendance write) fire-and-forget POST `evaluateAttendanceRules` with
**attendance row ids only**; the function re-derives swimmer/group/markedBy/
practice_date from PG (the processSession precedent ‚Äî zero trust in client
payload), evaluates, upserts. Scheduled sweep re-evaluates attendance rows
with `updated_at > now()‚àí10min` every 5 minutes (idempotent upsert absorbs
overlap; no schema marker needed). Same `x-process-secret` gate + env lines
banked at F.
(b) Scheduled-only evaluation (no kick): simpler, but trades today's
near-instant firing for up-to-N-minute latency.
(c) PG trigger evaluating in SQL: pgTAP-provable, but forks the shared
evaluation module into the database and rewrites ratified business logic ‚Äî
the A3/D-D5 precedent covers atomicity seams, not rule engines.

**D-G2 ‚Äî Coach push transport (and the RG-3 seam).**
(a) **RECOMMENDED ‚Äî G ships coach notifications in-app only; no push wiring.**
Writers (evaluator/digest) write `in_app_notifications` directly (service
role, upsert); **no `notification_jobs` writes; the Deno sender is not
touched**. `registerForPushNotifications` re-points its storage to
`push_tokens` (canonical parity, tests move) but remains uncalled ‚Äî exactly
today's effective behavior (RG-2: zero pushes delivered today). The seam
disappears instead of being patched; zero untestable surface. Coach push
wiring becomes an explicit post-cutover product line (sketch: jobs +
`skip_in_app` flag or a jest-tested Expo transport on the functions side).
(b) Wire push now via jobs + patch the sender to skip in-app inserts for
flagged jobs: one pipeline, but modifies untested Deno (RG-5) ‚Äî can't meet
the bar, permanent flagged risk.
(c) Send Expo pushes directly from the functions-side writers now: jest-tested
logic, but the transport call is mock-only (the Vertex class of risk) and
"push on first insert only" needs careful upsert discipline. Viable if you
want coach push to exist before cutover; honest cost is one more mock-only
external call.

**D-G3 ‚Äî OD-4: digest enumeration + pref home.**
(a) **RECOMMENDED ‚Äî enumerate `profiles` WHERE staff role set (the is_staff
set), gated by a new `notification_preferences.digest_enabled BOOLEAN NOT NULL
DEFAULT TRUE` (00008).** Faithful to today (every coach signup writes
default-true; no UI changes it), gives the pref a canonical home before the
AuthContext writer retires at cutover (the column default replaces the signup
write), and satisfies RG-8 provably: recipients come from the staff role set
by construction, never from prefs rows. **Edge-case flip to ratify:** today a
coach with a MISSING pref map is skipped; with a default-true column an absent
row means enabled. The old "skips coaches without dailyDigest preference" pin
is deleted by name (its subject ‚Äî the Firestore prefs map ‚Äî is deleted) and
replaced by an absent-row-means-enabled pin.
(b) No pref; digest to all staff unconditionally: loses the stored bit and the
future toggle's home.
(c) JSONB prefs blob: over-general, no canonical precedent.

**D-G4 ‚Äî `ai_drafts_ready` routing (banked: "G's whole").**
(a) **RECOMMENDED ‚Äî re-bank as a named post-launch product line.** No writer
has ever existed; migration phases reproduce today's behavior ("migrate
faithfully, do not editorialize"). G lands the category in the CHECK domain,
so wiring later is a one-line writer + tests.
(b) Wire it now: `processAudio/VideoSession` insert an in_app row (category
`ai_drafts_ready`, recipient = session coach's user_id) when drafts reach
review. ~15 lines + 2 jest pins, but it is genuinely NEW behavior landing
mid-migration.

**D-G5 ‚Äî Realtime publication (RG-6, cross-phase).**
(a) **RECOMMENDED ‚Äî land it as code in 00008:** idempotent DO-block
`ALTER PUBLICATION supabase_realtime ADD TABLE ‚Ä¶` for all 14 subscribed
tables (swimmers, attendance, swim_results, goals, group_notes, swimmer_notes,
swimmer_voice_notes, swimmer_coach_profile, audio_sessions,
audio_session_drafts, video_sessions, video_session_drafts + G's
notification_rules, in_app_notifications), guarded against already-present
membership; pgTAP proves `pg_publication_tables` membership. Realtime
postgres_changes respects RLS, so publication membership widens nothing.
(b) Runbook/cutover line only (dashboard config): leaves the gap latent and
unprovable.

**D-G6 ‚Äî Scheduling the dormant canonical pipeline (sender + cleanup-tokens).**
(a) **RECOMMENDED ‚Äî G changes no Deno code; the cutover runbook gains a line:**
"schedule `send-notification` + `cleanup-tokens` (Supabase cron) at cutover
staging; verify one job drains end-to-end (in-app rows + push) on the
throwaway project before flipping". Status quo is dormant-by-design pre-launch.
(b) pg_cron + pg_net in a migration: code-first but pulls the service key into
DB settings and adds extension surface ‚Äî poor trade pre-launch.

## ¬ß6 Red-team (attacking my own recommendations)

- **D-G2(a) means coaches still get zero pushes after G.** That IS today's
  effective behavior, but if you think of push as part of the feature's
  promise, (c) is the honest alternative ‚Äî its cost is one more mock-only
  external transport (the accepted Vertex class), not architecture.
- **Kick-from-checkOut**: re-evaluating on checkout can fire a rule created
  after check-in ‚Äî identical to today's onDocumentWritten behavior. Faithful.
- **batchCheckIn** sends ONE kick with N ids, not N kicks; the sweep catches
  dropped kicks (at-least-once), upsert makes repeats invisible.
- **Expression-index ON CONFLICT (RG-9)** is the likeliest silent-failure
  point; jest mocks cannot prove PG semantics, so pgTAP 011 carries the
  upsert proof (DO-block double-call ‚Üí single row, updated fields).
- **The digest count changes meaning by exactly one ratified step**: it adopts
  D-C5 absent-exclusion (your standing G rule). An absent-marked row (possible
  since C; impossible in Coach Firestore) no longer counts as "attended".
  Pinned in the new tests; old pin's subject (Firestore docs) is deleted.
- **Missing-pref flip (D-G3(a))** is a real behavior change at one edge ‚Äî
  named deletion + new pin, ratify it explicitly.
- **Publication DO-block** must tolerate tables already in the publication
  (local stack may pre-add some) ‚Äî membership-checked ADDs, idempotent under
  `db reset`.
- **If notification_rules realtime needs RLS context**, postgres_changes
  enforces RLS per-subscriber ‚Äî staff-only rules emit only to staff. No
  widening through the publication.
- **Digest enumerated from profiles role set**: a future role widening would
  widen digest recipients silently ‚Äî the pgTAP wall on rules/in_app doesn't
  guard enumeration (it's service-role). Jest pins the role filter string.
